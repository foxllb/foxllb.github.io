相关原理解析

写时复制

  • fork: 用于创建子进程, 返回两个pid, 接收pid=0的进程为子进程, 父进程接收的pid为子进程的pid

  • exec: exec组函数(execl, execlp, execle, execv, execvp, execvpe)使用新的进程映像取代现有进程映像

  • 在linux中通常使用fork + exec运行新进程, 那么如果fork创建子进程时将父进程内存全部复制, 但在exec运行时, 新的进程内存映像会替换原有进程内存映像, 即fork的复制操作冗余, 对系统效率有极大耗损, 与之相对应的改进方案就是写时复制

  • fork的写时复制机制实现原理: 子进程几乎完全复制父进程的task_struct(pid除外), 从内存管理的角度上来讲, 父子进程可以访问相同的共享区域(共享页被标记为写时复制), 当进程进行写操作时(并非所有共享页都可以进行写时复制, 比如code segment不可被修改, 就没有使用写时复制的必要, 父子进程直接共享code segment), 会复制对应的共享页, 在此基础上进行修改

缺页中断

缺页中断解释起来需要很大的篇幅，留待以后总结。

write流程解析(默认读者了解缺页中断原理)

假设程序通过mmap(私有映射+read_only)映射某只读文件后使用write系统调用向映射区域写数据

  1. __access_remote_vm

程序进入__access_remote_vm后, 直接调用get_user_pages试图获得合法page

  1. __access_remote_vm -> __get_user_pages_locked

设置foll_flag, 本流程中foll_flag被设置为FOLL_GET| FOLL_WRITE| FOLL_FORCE

  1. __access_remote_vm -> __get_user_pages_locked -> __get_user_pages.retry(第一次进入retry)

程序第一次执行到follow_page_mask.retry时携带标志位FOLL_GET| FOLL_WRITE| FOLL_FORCE, 使用follow_page_mask用于获得虚拟地址对应的page(因为文件使用mmap私有映射, 所以此时虚拟地址没有与真实物理地址关联, follow_page_mask返回NULL)

  1. __access_remote_vm -> __get_user_pages_locked -> __get_user_pages.retry -> faultin_page

faultin_page主动触发写中断错误, 进入handle_mm_fault

  1. __access_remote_vm -> __get_user_pages_locked -> __get_user_pages.retry -> faultin_page -> handle_mm_fault -> handle_pte_fault

第一次中断是因为虚拟地址没有和物理地址关联, 所以pte无效, 满足!pte_present(entry)&&!pte_none(entry), mmap实现了私有文件映射不满足匿名页面条件所以进入do_fault

  1. __access_remote_vm -> __get_user_pages_locked -> __get_user_pages.retry -> faultin_page -> handle_mm_fault -> handle_pte_fault -> do_fault

以上流程中提到过, 本次缺页中断为主动触发的写中断错误, 且为mmap私有映射所以满足(flags & FAULT_FLAG_WRITE)&&!(vma->vm_flags & VM_SHARED), 进入do_cow_fault

  1. __access_remote_vm -> __get_user_pages_locked -> __get_user_pages.retry -> faultin_page -> handle_mm_fault -> handle_pte_fault -> do_fault -> do_cow_fault

do_cow_fault为new_page分配page后, 利用__do_fault将page_cache读入fault_page, 然后把文件内容复制到new_page, 使用do_set_pte建立虚拟地址与物理地址页表关联后, 释放fault_page

  1. __access_remote_vm -> __get_user_pages_locked -> __get_user_pages.retry(第二次进入retry)

第一次中断结束后返回__get_user_pages.retry再次执行follow_page_mask, 标志位FOLL_GET| FOLL_WRITE| FOLL_FORCE不变, 由上述流程得到的pte满足present, dirty, readonly但是不可写, 所以follow_page_mask再次失败返回NULL, 再次主动触发缺页中断

  1. __access_remote_vm -> __get_user_pages_locked -> __get_user_pages.retry -> faultin_page -> handle_mm_fault -> handle_pte_fault(代码见5)

本次满足pte_present(entry)&&!pte_none(entry), 进入do_swap_page进入页交换

  1. __access_remote_vm -> __get_user_pages_locked -> __get_user_pages.retry -> faultin_page -> handle_mm_fault -> handle_pte_fault -> do_swap_page -> wp_page_reuse

在页交换阶段同样会进行maybe_mkwrite, 但是无法写入write标志位, 同时wp_page_reuse会 return VM_FAULT_WRITE

  1. __access_remote_vm -> __get_user_pages_locked -> __get_user_pages.retry -> faultin_page(代码见4)

由wp_page_reuse开始return VM_FAULT_WRITE, 一直返回到达faultin_page, 满足如下条件, 取消FOLL_WRITE标志(表示已经完成了写时复制处理工作)

  1. __access_remote_vm -> __get_user_pages_locked -> __get_user_pages.retry(第三次进入retry)

只读调用follow_page_mask成功, 返回正常的page

  1. __access_remote_vm

返回__access_remote_vm, 由kmap映射该page并写入(并不会影响磁盘文件)

madvise

  • madvise 故名思意, 针对内存的建议, 在用户态中使用

  • int madvise(void *addr, size_t length, MADV_DONTNEED);

    • 从addr开始, 长度为length的区域空间在未来将不会被使用, 释放对应的物理页(会直接导致pte无效)

exp流程解析

write流程解析中的3中提到cond_resched();会让出cpu, 这意味着retry执行会存在间隙, 那么如果write流程将要执行13, 此时madvise使得pte无效那么系统执行流程如下:

  1. madvise执行完毕后, pte无效, 此时继续执行retry.follow_page_mask, 但是pte无效所以返回NULL, 第三次主动触发缺页中断(建立虚拟地址与物理页的联系, 重建pte)

  2. 进入do_fault后因为此次缺页中断没有FOLL_WRITE限制, 所以触发了读缺页错误(page cache读取文件内容并return 0, madvise释放的是cow产生的匿名页面(new_page))

  1. 第四次执行retry后, raetry.follow_page_msk返回正常page(指向page cache)

  2. kmap重新映射, 写入内容, 设置dirty, 最终更新内容被更新到磁盘文件, 完成只读文件写操作

patch分析

  • patch链接: commit = 19be0eaffa3ac7d8eb6784ad9bdbc7d67ed8e619

patch方案

  1. 添加FOLL_COW标志, 表示已完成写时复制用于取代原有的写时复制后取消FOLL_WRITE机制

  2. follow_page_mask中添加can_follow_write_pte用于检测pte检测是否同时满足((flags & FOLL_FORCE) && (flags & FOLL_COW) && pte_dirty(pte))

发表于 2021-03-20 更新于 2021-04-26 分类于 Linux用户态

CVE-2019-14287(sudo 逻辑漏洞提权)

适用范围

  • sudo版本 < 1.8.28

环境复现

  • sudo源码下载
  • 编译(make uninstall时报错, 建议提前拍摄虚拟机快照, 以便复现cve后继续正常使用虚拟机)

验证提权思路

相关原理

ruid, euid, suid

  • ruid(real uid), 用户实际id

  • euid(effective uid), 用户当前的有效id, 决定用户当前状态对资源的访问权限

  • suid(set uid ID upon execution) 指文件权限对外开放, 当其他用户运行此文件时, 拥有与文件拥有者相同权限

  • 举例:

    • root(uid = 0, euid = 0) 设置 chmod +s file, 则file文件被标志为suid

    • povcfe(uid = 1000, euid = 1000) 执行file文件, 此时权限: povcfe(uid = 1000, euid = 0), povcfe可以通过file访问root资源

setresuid

  • setresuid可以被理解成setruid_euid_suid, 即同时设置ruid, euid, suid

  • setresuid(new_ruid, new_euid, new_suid) 执行成功条件:

    • old_ruid, old_euid, old_suid任一项为0, setresuid必执行成功, ruid, euid, suid, 被同时修改为new

    • old_ruid, old_euid, old_suid 中的任一个id值必须在 ruid, euid, suid中出现

  • 特殊情况setresuid(-1, -1, -1), 不更改用户权限(原因如下):setresuid -> setresuid32 -> sys_setresuid

漏洞分析

patch 分析

  • patch链接, 从中定位到与当前cve相关的描述: Disallow id -1, which means “no change”, 断定此处为对于漏洞的修补, 该修复限定(llval != -1 || llval != (id_t)UINT_MAX)
  • patch位于sudo_strtoid_v1函数, 回到1.8.25版本sudo源码的sudo_strtoid_v1, 查看函数逻辑

验证分析

  • 跟踪系统调用setuid, setresuid(注意最后两次系统调用setuid(0) + setresuid), 设置uid=0成功后, -u#-1对应的setresuid(-1, -1, -1)保持原权限(0, 0, 0)

小结

  • strace 跟踪系统调用 + systemtap 跟踪内核事件

  • 代码审计非常重要, 现有的fuzz是很难发现该漏洞的

hijack_prctl篇

  • prctl函数: 用户态函数，可用于定制进程参数，非常适合和内核进行交互

    3. security_task_prctl执行hook.task_prctl

  • 攻击流程:

    1. 劫持hook.task_prctl为目标函数地址(poweroff_work_func)
    2. 修改poweroff_cmd为目标指令

  • gdb 执行 p poweroff_cmd，获得poweroff_cmd真实地址

  • 获得hook.prctl，poweroff_cmd与内核基地址固定偏移

Linux内核提权总结----任意读写(2)

发表于 2021-01-21 更新于 2021-04-26 分类于 Linux内核态

hijack_vdso篇

  • vdso: 内核实现的一个动态库，存在于内核，然后映射到用户态空间，可由用户态直接调用

    • 内核中的vdso如果被修改，那么用户态空间的vdso也会同步被修改

  • 攻击流程

    1. (内核任意代码执行漏洞)内核调用set_memory_rw 函数修改内核vdso页面属性，使得用户态可以直接修改vdso，劫持vdso为shellcode，触发条件同1
    2. (内核任意读写漏洞)内核修改内核vdso数据，写入shellcode，使得用户态vdso中函数被劫持，当高权限进程调用vdso中特定函数时，触发shellcode，本篇只讲解攻击流程2

一. 利用步骤

1. 定位内核态vdso位置

  • vdso中存在一些比较有特点的字符串，比如”gettimeofday”，在拥有任意读漏洞的前提下，从0xffffffff80000000(开启kaslr后内核基地址在此地址基础上往上偏移)开始按页搜索内存

  • 如果内存其他地方存在”gettimeofday”字符串，且出现在vdso之前，则会返回错误地址。所以在内存搜索时，应以返回地址为起始，使用gdb dump 0x2000 内存，使用ida查看是否是vdso，如果不是，可以忽略这个错误地址，继续向下搜索。

  • 得到真正的vdso后，查看”gettimeofday”与vdso起始地址的偏移，后续匹配vdso时，加上这个偏移条件

2. 向内核中vdso写入shellcode

  • shellcode功能: 判断进程是否具有root权限，如果成立，则开辟新进程用来反弹root shell

  • shellcode写入位置: 定位vdso上某函数，比如gettimeofday函数，使用shellcode覆盖

  • shellcode触发条件: 当高权限进程调用gettimeofday函数时，自动执行shellcode 反弹root shell

Linux内核提权总结----ROP(1)

发表于 2021-01-01 更新于 2021-05-11 分类于 Linux内核态

基础ROP篇(linux 5.0.21)

内核提权与用户态攻击的区别

  • 攻击流程

    • 用户态攻击: 执行 system(“/bin/sh”) 获得shell
    • 内核提权:
      1. 内核执行 commit_creds(prepare_kernel_cred(0)) 使进程获得root权限
      2. 用户态进程执行system(“/bin/sh”) 获得root权限 shell

  • 理解难点

    • 内核rop链构造
    • 用户态进程与内核之间的切换

一. 漏洞分析 (建议初学者先了解基础的驱动程序知识)

  1. 查看驱动安全机制

    • 发现开启canary和NX

  2. 查看qemu启动脚本boot.sh

    • boot.sh

  3. 查看开机自启脚本

    • rcS

  4. 审计代码，可以发现dangerous函数存在明显的栈溢出漏洞(降低难度直接泄漏canary)

  5. 利用思路

    • Canary通过printk泄露
    • NX使用rop绕过
    • 读取 /tmp/kallsyms的startup_64(获得内核代码加载基地址，用于定位ROPgadget在内存中的真实地址)
    • 读取 /tmp/kallsyms的commit_creds, prepare_kernel_cred，获得目标内核函数地址

二. ROP链构造

  • 内核执行commit_creds(prepare_kernel_cred(0))

    1. 通过/tmp/kallsyms获得 commit_creds() prepare_kernel_cred() 函数地址

    1. 执行 cat tmp/kallsyms | grep startup_64, 如果startup_64不等于0xffffffff81000000, 即内核开启kalsr防护, startup_64与0xffffffff81000000 的差值即为内核基地址偏移(很多发行版默认不打开kalsr, 利用就比较简单了)

    1. 利用ROPgadget获得vmlinux的所有gadget片段(如果效果不理想可换用ropper) 根据所需在rop_gadget中搜索(因为程序开启kalsr，所以 真实地址 = rop_gadget地址 + 基地址偏移)

  • 返回用户态

    1. 进入内核前，保存用户态数据

    2. 内核切换到用户态

    • 执行swapg 恢复GS值(GS通过宏实现, 并不保存在gs寄存器)
    • 执行iretq 恢复用户态数据(或者执行sysretq 只需提供rip)
      • 需要提前在栈上填充用户态数据(rip,cs,rflags,sp,ss)
      • 填充的用户态数据应在用户态切换到内核态前保存(内联汇编)

  • ROP 链构造

    1. commit_creds() 接收 prepare_kernel_cred(0)的返回值作为参数，即rdi <- rax 时需要灵活变通(手动搜索rop_gadget文件)

五. 踩坑记录

  1. 高版本linux canary存在\x00截断(低版本没有)
  2. 使用ROPgadget获得的gadget片段有可能存在于vmlinux的数据区 (不管是代码还是数据在内存中都只是一堆二进制数字), 所以如果内核开启NX保护, 那么内核数据区的gadget就不能用于ROP链构造

    • 解决方案: 将qemu的cpu启动参数由kvm64切换为qemu64
  4. ROPgadget 获得的代码片段中没有iretq和sysretq
    • 原因：ROPgadget工具获得的代码片段作用于程序流程控制，即ret, call 等元素必须存在，但是内核态返回用户态只需执行 iretq指令，与其后接的汇编无关，由此看来，使用ROPgadget工具获得的内容是少于需求的，所以有可能iretq,sysretq, 不存在于获得的代码片段中
    • 解决方案：
    1. python 进行iretq编码匹配
    2. ida 解析vmlinux 查找iretq指令
